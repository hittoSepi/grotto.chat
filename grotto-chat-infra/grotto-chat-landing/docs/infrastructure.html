Infrastructure & Self-Hosting

Everything you need to run your own MobulaDock server — from a Raspberry Pi to a cloud VPS.

Overview

Running your own MobulaDock server gives you full control over your messaging infrastructure. The server is a lightweight C++20 binary that relays encrypted messages between clients without ever seeing their content. To get started, you need three things:

  • A Linux machine — physical or virtual, aarch64 or x64
  • A domain name (optional) — makes it easier for others to connect
  • A TLS certificate — all connections require TLS 1.3

Here is a typical deployment topology:

Deployment Architecture
InternetClients connect here
→
FirewallPort 6697
→
Reverse Proxy(optional) nginx
→
MobulaDock ServerTLS Relay
↓
SQLite DBMetadata & offline msgs

The reverse proxy layer is optional. If you terminate TLS directly at the MobulaDock server (the common setup), you can skip it entirely. A reverse proxy is useful if you are hosting multiple services on the same machine or need advanced load balancing.

System Requirements

MobulaDock is designed to be lightweight. The server binary is statically linked and has minimal dependencies beyond the OS kernel.

Resource Minimum Recommended
CPU 1 core 2+ cores
RAM 256 MB 512 MB+
Storage 100 MB + database 1 GB+ (room for logs and backups)
OS Linux (aarch64 or x86_64)
Network Stable connection, static IP or DDNS
Raspberry Pi Friendly

MobulaDock runs great on a Raspberry Pi 4 or newer. The aarch64 build is a first-class target, and the low resource footprint means you can run it alongside other services without issue.

TLS Setup

All MobulaDock connections use TLS 1.3. This is not optional — the server will not accept plaintext connections. TLS protects the metadata layer (who is talking to whom) while the Signal Protocol protects message content end-to-end.

Option A: Let's Encrypt (recommended)

If you have a domain name pointing to your server, Let's Encrypt provides free, automatically renewing certificates. Install certbot and run the standalone challenge:

bash 
# Install certbot
sudo apt update
sudo apt install certbot

# Obtain certificate (stop any service on port 80 first)
sudo certbot certonly --standalone -d your-domain.com

# Certificates are saved to:
#   /etc/letsencrypt/live/your-domain.com/fullchain.pem
#   /etc/letsencrypt/live/your-domain.com/privkey.pem

Point your server.toml at the generated files:

toml 
[tls]
cert = "/etc/letsencrypt/live/your-domain.com/fullchain.pem"
key  = "/etc/letsencrypt/live/your-domain.com/privkey.pem"

Certbot sets up automatic renewal by default. Restart the MobulaDock server after each renewal to pick up the new certificate, or use a post-renewal hook:

bash 
# /etc/letsencrypt/renewal-hooks/post/restart-mobuladock.sh
#!/bin/bash
systemctl restart mobuladock

Option B: Self-Signed Certificate

For testing or private networks where you control all clients, a self-signed certificate works fine.

bash 
# Generate a self-signed certificate valid for 365 days
openssl req -x509 -newkey ec -pkeyopt ec_paramgen_curve:prime256v1 \
  -keyout server.key -out server.crt -days 365 -nodes \
  -subj "/CN=mobuladock-server"

# Move to a permanent location
sudo mkdir -p /etc/mobuladock/tls
sudo mv server.crt server.key /etc/mobuladock/tls/
sudo chmod 600 /etc/mobuladock/tls/server.key
Trust Warnings

Clients connecting to a server with a self-signed certificate will see trust warnings. Each user will need to manually accept the certificate or install it as a trusted root on their device. For public-facing servers, use Let's Encrypt instead.

Public Server Directory

MobulaDock includes an optional public server directory — a central Node.js service where servers can register themselves so users can discover and connect to them from within their client. Listing your server is entirely optional; private servers work perfectly without it.

How It Works

When directory registration is enabled, your server periodically sends a heartbeat to the directory service. The directory maintains a list of active servers along with basic metadata (name, description, user count). Clients can browse this list to find servers to join.

Configuration

Add a [directory] section to your server.toml:

toml 
[directory]
enabled     = true
url         = "https://directory.mobuladock.org"
name        = "My MobulaDock Server"
description = "A friendly community server"
public_host = "mobuladock.example.com"
public_port = 6697
Private Servers

If you omit the [directory] section or set enabled = false, your server will not appear in any public listing. Users can still connect directly by entering your hostname and port. This is the default behavior.

Firewall & Networking

The MobulaDock server listens on port 6697 by default (the standard IRC-over-TLS port). You need to allow inbound TCP traffic on this port.

UFW (Uncomplicated Firewall)

bash 
# Allow MobulaDock traffic
sudo ufw allow 6697/tcp comment "MobulaDock"

# Verify the rule
sudo ufw status

# If you also need certbot HTTP challenge
sudo ufw allow 80/tcp comment "Certbot"

Reverse Proxy with nginx

If you want to put MobulaDock behind nginx (for example, to share port 443 with a website), you can use a TCP stream proxy. Note that this is a stream block, not an http block, because MobulaDock uses raw TLS sockets rather than HTTPS.

nginx 
# /etc/nginx/nginx.conf (or a file in stream.d/)
stream {
    upstream mobuladock_backend {
        server 127.0.0.1:6697;
    }

    server {
        listen 6698 ssl;
        proxy_pass mobuladock_backend;

        ssl_certificate     /etc/letsencrypt/live/your-domain.com/fullchain.pem;
        ssl_certificate_key /etc/letsencrypt/live/your-domain.com/privkey.pem;
        ssl_protocols       TLSv1.3;
    }
}
NAT / Home Networks

If your server is behind a home router (NAT), you will need to set up port forwarding. Forward external port 6697 (TCP) to your server's local IP address. Consider using a DDNS service if your public IP changes frequently.

Monitoring & Logs

The MobulaDock server logs to stdout by default, making it easy to integrate with any log management solution. For production deployments, running the server as a systemd service is recommended.

systemd Service File

ini 
# /etc/systemd/system/mobuladock.service
[Unit]
Description=MobulaDock Server
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=mobuladock
Group=mobuladock
WorkingDirectory=/opt/mobuladock
ExecStart=/opt/mobuladock/mobuladock-server --config /etc/mobuladock/server.toml
Restart=on-failure
RestartSec=5
StandardOutput=journal
StandardError=journal

# Security hardening
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
ReadWritePaths=/var/lib/mobuladock

[Install]
WantedBy=multi-user.target
bash 
# Enable and start the service
sudo systemctl daemon-reload
sudo systemctl enable mobuladock
sudo systemctl start mobuladock

# View logs
sudo journalctl -u mobuladock -f

Version Verification

Use the --version flag to check which build is deployed. The output includes the git commit hash and build timestamp, which helps verify that the correct version is running after a deployment.

bash 
$ ./mobuladock-server --version
mobuladock-server v0.15.0 (abc1234, built 2026-03-13T10:30:00Z)

Log Levels

Level What It Shows
error Critical failures: TLS errors, bind failures, database corruption
warn Recoverable issues: client disconnects, malformed messages, auth failures
info Normal operation: connections, joins, parts, server start/stop (default)
debug Verbose output: message routing, session state, protocol details

Protocol Handler

MobulaDock supports the ircord:// URI scheme, allowing users to click a link and connect directly to a server. The format is:

text 
ircord://hostname:port

For example: ircord://chat.example.com:6697

Registering on Desktop (Linux)

Create a .desktop file for the protocol handler:

ini 
# ~/.local/share/applications/mobuladock-handler.desktop
[Desktop Entry]
Name=MobulaDock
Exec=/usr/local/bin/mobuladock-client --connect %u
Type=Application
MimeType=x-scheme-handler/mobuladock;
NoDisplay=true
bash 
# Register the handler
xdg-mime default mobuladock-handler.desktop x-scheme-handler/mobuladock

Registering on Desktop (Windows)

The Windows installer registers the protocol handler automatically. For manual registration, add a registry key:

text 
HKEY_CLASSES_ROOT\mobuladock
  (Default) = "URL:MobulaDock Protocol"
  "URL Protocol" = ""
  shell\open\command
    (Default) = "C:\Program Files\MobulaDock\mobuladock-client.exe" "--connect" "%1"
Android Support

The Android client handles ircord:// links natively through intent filters. No additional setup is required on mobile devices.

Backup & Recovery

MobulaDock stores all persistent data in a small number of files, making backups straightforward. Here is what you should back up:

Item Location Notes
Configuration /etc/mobuladock/server.toml Server settings, directory config
TLS Certificates /etc/mobuladock/tls/ or Let's Encrypt path Can be regenerated, but saves downtime
Database /var/lib/mobuladock/mobuladock.db SQLite file — user accounts, offline messages, metadata
Plugin Data /var/lib/mobuladock/plugins/ Plugin configurations and persistent storage

Backup Script

The database is a single SQLite file, so a simple file copy is all you need. Here is a basic backup script you can run via cron:

bash 
#!/bin/bash
# /opt/mobuladock/backup.sh — run daily via cron
set -euo pipefail

BACKUP_DIR="/var/backups/mobuladock"
DATE=$(date +%Y-%m-%d)
DEST="${BACKUP_DIR}/${DATE}"

mkdir -p "${DEST}"

# Back up configuration
cp /etc/mobuladock/server.toml "${DEST}/"

# Back up database (use sqlite3 .backup for consistency)
sqlite3 /var/lib/mobuladock/mobuladock.db ".backup '${DEST}/mobuladock.db'"

# Back up plugin data
if [ -d /var/lib/mobuladock/plugins ]; then
    cp -r /var/lib/mobuladock/plugins "${DEST}/"
fi

# Remove backups older than 30 days
find "${BACKUP_DIR}" -maxdepth 1 -type d -mtime +30 -exec rm -rf {} \;

echo "Backup complete: ${DEST}"
bash 
# Add to crontab — run daily at 3 AM
sudo crontab -e
# Add this line:
0 3 * * * /opt/mobuladock/backup.sh >> /var/log/mobuladock-backup.log 2>&1
Recovery

To restore from a backup, stop the server, copy the backed-up files to their original locations, and restart. The SQLite database requires no special import process — just replace the file.
