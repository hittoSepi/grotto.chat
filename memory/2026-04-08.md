# 2026-04-08 - File Transfer Hardening, Files Panel, and Quotas

## Implemented
- Hardened desktop/server file transfer flow end-to-end.
- Completed checksum-aware upload/download path:
  - client sends chunk checksums
  - server validates chunk checksums when present
  - server computes final SHA-256 and returns it in `MT_FILE_COMPLETE`
  - client verifies completed downloads against `file_checksum`
- Replaced antivirus stub with real ClamAV/`clamd` integration and fail-closed behavior when configured but unavailable.
- Fixed several protocol/runtime bugs in file transfer:
  - upload chunk framing too large for wire limits
  - download chunk framing too large for wire limits
  - download path needed server-side rechunking from DB chunks into safe wire frames
  - file chunk traffic incorrectly hit chat rate-limit path
  - concurrent TLS writes during download caused `bad length (SSL routines)` disconnects
- Locked down file download authorization:
  - DM file: sender or recipient only
  - channel file: current channel member only
- Added server/client file policy bootstrap:
  - `max_upload_bytes`
  - MIME allow/block rules
  - later extended with server/user storage quota values
- Added file list protocol and live files browser:
  - `MT_FILE_LIST_REQUEST`
  - `MT_FILE_LIST_RESPONSE`
  - `MT_FILE_CHANGED`
- Built a general secondary side-panel slot in the client UI and used it for a files panel.
- Files panel behavior now supports:
  - `F3` to open
  - auto-refresh on file events
  - resize like user list
  - arrow navigation
  - `Enter` to download
  - double-click to download
  - `r` to refresh
  - `o` to open local downloads folder
- Improved files panel presentation:
  - newest files first
  - clearer metadata (size, timestamp, sender)
  - clearer key hints
- Fixed the long-standing local slash-command feedback/rendering bug after `/help`.
- Added server storage quotas:
  - `max_total_storage_bytes`
  - `max_user_storage_bytes`
  - quotas reserve storage at upload start, including incomplete non-expired uploads
- Extended `MT_FILE_POLICY` so clients now receive advertised quota limits too.
- Client upload preflight now blocks files that are individually larger than the advertised server/user quota limits.

## Important Design/Behavior Notes
- Server-side file storage is still SQLite-backed, not filesystem-backed:
  - file metadata in `files`
  - chunk data in `file_chunks`
- Uploads are target-scoped in metadata:
  - channel upload => `channel_id`
  - DM upload => `recipient_id`
- Authorization is enforced on download, not just upload metadata.
- Quota bootstrap does not expose current remaining free space yet.
  - Client can block "always impossible" uploads.
  - Server remains the final authority for real quota enforcement.
